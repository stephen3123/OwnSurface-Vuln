OwnSurface — Complete Platform Documentation

What Is OwnSurface?

OwnSurface lets you scan any website, discover its full technology stack, test your own domains for security vulnerabilities, and monitor everything from one platform. Point it at any URL and get instant intelligence — from what frameworks a site uses to whether it has SQL injection flaws, exposed secrets, or misconfigured cloud buckets.

For your own verified domains, OwnSurface goes deeper: run offensive penetration tests (SQLi, XSS, CSRF, SSRF, auth bypass), crawl every page with a headless browser, map your full attack surface, monitor uptime and SSL certificates, check compliance, and generate security reports — all without installing 17 separate tools.

Key Capabilities

  • Website Intelligence — Scan any URL to detect tech stack, company info, SEO, traffic, social links, and business signals. Replaces BuiltWith, Wappalyzer, SimilarWeb, Clearbit, and SecurityHeaders in one scan.
  • Vulnerability Detection — 29 scanner modules check for exposed files, CVEs, insecure headers, CORS misconfigurations, DNS security issues, leaked secrets, and more.
  • Offensive Security Testing — 22 penetration testing modules powered by industry tools (sqlmap, dalfox, subfinder, nuclei, ffuf, naabu, gitleaks) with automatic safety controls. Test for SQL injection, XSS, CSRF, SSRF, JWT flaws, auth bypass, path traversal, IDOR, subdomain takeover, open ports, cloud bucket misconfigs, and more.
  • Attack Surface Mapping — 4-tier audit: passive reconnaissance, active probing, Nuclei vulnerability scanning, and offensive testing with AI-generated summary.
  • Monitoring — Uptime checks, SSL certificate expiry tracking, and Core Web Vitals performance monitoring with alerts.
  • Lead Intelligence — Search companies by tech stack, industry, employee count, and location. Export leads as CSV. Clearbit alternative built into every scan.
  • AI Visibility — Check how your brand appears in AI model responses (ChatGPT, Claude, etc.).
  • Compliance Auditing — Check domains against GDPR, CCPA, HIPAA and other regulations.
  • CI/CD Integration — SARIF output, --fail-on severity gates, and a GitHub Action to block deployments on critical findings.

How a Scan Works (End-to-End Flow)

  1. User submits a URL from any surface (dashboard, extension, CLI, MCP, or API)
  2. API server receives the request at POST /api/v1/scan
  3. API checks Dragonfly cache using a SHA-256 hash of the normalized URL
  4. If cached (within 24 hours), the cached result is returned immediately
  5. If not cached, the API publishes a scan.request message to NATS with the URL and hash
  6. A worker picks up the message from the NATS queue (load-balanced across workers)
  7. The worker launches a Playwright browser and runs all 29 scanner modules in parallel
  8. Sequential post-processing runs: CVE matching (needs tech stack), Lighthouse scores (needs performance data), carbon footprint (needs page weight + traffic), AI summary (needs all results)
  9. The worker publishes the complete result to scan.result.{hash} on NATS
  10. The API receives the result, caches it in Dragonfly with a 24-hour TTL, and stores it in PostgreSQL
  11. The result is returned to the user

The same flow applies to all surfaces. The CLI, MCP, extension, and dashboard all call the same API endpoint. The only difference is how the result is displayed.

Safety Controls

  • Domain verification checked at both API and worker level (defense-in-depth)
  • Max request rate: 20 requests per second
  • Max total requests: 5,000 per scan
  • Auto-stop on: WAF detection (403/429), target response time above 15 seconds, 10 consecutive errors
  • Per-domain locking: only 1 concurrent offensive scan per domain
  • Rate limiting: conservative (500ms), moderate (200ms), aggressive (50ms) delay between requests
  • All tools spawned via execFile with 120-second timeout and 50MB stdout cap
  • Input sanitization blocks private IP ranges, shell metacharacters, and path traversal

Worker Concurrency

  • Configurable via WORKER_CONCURRENCY environment variable (default: 3)
  • Controls the Playwright browser pool size
  • NATS queue group scan-workers distributes requests across multiple worker instances
  • Each subscription worker has auto-restart with exponential backoff (2s to 30s)
  • Graceful shutdown: SIGINT/SIGTERM drains NATS connections and closes browser pools

Docker Build

The worker uses a multi-stage Docker build (Dockerfile.offensive):

  • Stage 1 (Go): Compiles 10 security tools (nuclei, subfinder, httpx, katana, naabu, dalfox, ffuf, gitleaks, subjack, s3scanner)
  • Stage 2 (Python): Installs 7 tools (sqlmap, xsstrike, xsrfprobe, graphql-cop, wfuzz, jwt_tool, SSRFmap)
  • Stage 3 (Alpine): Clones PayloadsAllTheThings and curated SecLists subset (~200MB instead of full 1GB+)
  • Stage 4 (Runtime): Node.js 22 slim with Playwright Chromium, all tools copied from previous stages

Tools degrade gracefully if missing: each scanner checks tool availability via isBinaryAvailable() and falls back to built-in payloads.

GitHub Actions Integration

OwnSurface provides a GitHub Action at .github/actions/xray-scan/ for CI/CD pipeline integration.

Usage

- name: Security Scan
  uses: ./.github/actions/xray-scan
  with:
    url: "https://myapp.com"
    api-key: ${{ secrets.OWNSURFACE_API_KEY }}
    fail-on: "critical"
    format: "sarif"

Outputs

  • findings-count: Total number of security findings
  • critical-count: Number of critical findings
  • high-count: Number of high findings
  • sarif-file: Path to SARIF output file (for GitHub Advanced Security upload)

How It Works

The action runs in a Docker container with @ownsurface/cli installed globally. It executes ownsurface scan with JSON output, parses finding counts, optionally generates a SARIF file, and exits with code 1 if findings at or above the --fail-on severity threshold.

Mascot: Ray the Stingray

OwnSurface's mascot is Ray, a stingray that represents "seeing through surfaces." Ray appears in the CLI with five states:

  • Idle: White eyes, shown on welcome screen with blinking animation
  • Blink: Eyes close briefly every 1.2-2 seconds (randomized for natural feel)
  • Scanning: Yellow eyes with beam ())) and arrows (>>>) — shown during active scans
  • Found: Red eyes with warning symbol — shown when vulnerabilities detected
  • Happy: Green smiling eyes with checkmark — shown on clean scan results
  • Compact: One-line face for scan output headers

The blinking animation only runs in TTY terminals. In piped/CI output, Ray is rendered as a static image.
